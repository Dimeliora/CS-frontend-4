# Бенчмарк: push/pop vs unshift/shift (packed vs holey)

## Условия эксперимента

| Параметр                   | Значение                            |
| -------------------------- | ----------------------------------- |
| Операций за замер          | 100 000                             |
| Итераций замера            | 5                                   |
| Прогрев JIT                | 3 запуска на массиве размером 100   |
| Размеры начальных массивов | 100, 1 000, 10 000, 100 000         |
| Режимы массивов            | packed (плотный), holey (с дырками) |
| Среда                      | Node.js (V8), macOS                 |

### Запуск

```bash
npx tsx benchmark.ts              # с прогревом JIT
npx tsx benchmark.ts --no-warmup  # без прогрева JIT
```

## Выводы

### 1. push/pop vs unshift/shift: разница в ~1 000 000 раз

Самый главный вывод — **колоссальная** разница в производительности:

- **`push`/`pop`**: ~0.1–0.7 мс на 100 000 операций
- **`unshift`/`shift`**: ~650–2050 мс на 100 000 операций

Это объясняется алгоритмической сложностью:

| Операция  | Сложность                | Почему                                                                                                            |
| --------- | ------------------------ | ----------------------------------------------------------------------------------------------------------------- |
| `push`    | **O(1)** амортизированно | Добавление в конец — просто запись в следующий слот. Иногда нужна реаллокация буфера, но амортизированно это O(1) |
| `pop`     | **O(1)**                 | Удаление с конца — уменьшение длины, без сдвига элементов                                                         |
| `unshift` | **O(n)**                 | Добавление в начало требует сдвига **всех** существующих элементов вправо                                         |
| `shift`   | **O(n)**                 | Удаление из начала требует сдвига **всех** элементов влево                                                        |

### 2. Влияние размера массива

- **`push`/`pop`** — время **не зависит** от начального размера массива. Результаты стабильны (~0.1–0.7 мс) для всех размеров от 100 до 100 000.
- **`unshift`/`shift`** — время **растёт** с увеличением размера массива:
  - 100 элементов: ~650–680 мс
  - 10 000 элементов: ~800–830 мс
  - 100 000 элементов: ~2000–2060 мс

### 3. Packed vs Holey: минимальная разница

Разница между packed и holey массивами оказалась **незначительной** для всех операций:

- Для `push`/`pop` разница в пределах погрешности измерений
- Для `unshift`/`shift` разница составляет менее 3%

### 4. Влияние прогрева JIT

Сравнение режимов с прогревом и без показало **минимальную разницу**:

| Операция (100K)  | С прогревом | Без прогрева | Разница        |
| ---------------- | ----------- | ------------ | -------------- |
| push [packed]    | 0.661 мс    | 0.487 мс     | ~0.2 мс        |
| pop [packed]     | 0.164 мс    | 0.160 мс     | ~0.004 мс      |
| unshift [packed] | 2056 мс     | 2045 мс      | ~11 мс (~0.5%) |
| shift [packed]   | 2022 мс     | 2000 мс      | ~22 мс (~1%)   |

### 5. Общий вывод

1. На больших массивах (в бенчмарке порог между 10К и 100К элементов) следует избегать частых вставок/удалений с начала, для таких ситуаций лучше подойдёт **связный список** или **двусторонняя очередь**
2. Для реализации очереди на больших данных лучше использовать **кольцевой буфер** или **два стека** вместо `shift`
3. **`push`/`pop`** — оптимальный выбор для стека (размер массива практически не сказывается на эффективности)
4. Дырки в массивах **не влияют** существенно на `push`/`pop`/`unshift`/`shift`, но могут замедлить итерацию

## Appendix (вопрос о влиянии разного рода разреженности массива на результат)

Есть ли резон делать более разреженный массив со случайным расположением?
Короткий ответ: **для данного бенчмарка — нет, существенного резона нет**

1. V8 классифицирует массивы бинарно: PACKED или HOLEY
В V8 нет градаций «немного holey» и «очень holey». Как только массив получает хотя бы одну дырку, он переходит в режим `HOLEY_SMI_ELEMENTS` / `HOLEY_ELEMENTS` — и обратно в PACKED уже не вернётся. Текущая реализация с i += 2 уже гарантированно переводит массив в HOLEY-режим.

2. Операции `push`/`pop`/`unshift`/`shift` не зависят от паттерна дырок
Эти операции работают с концами массива:
- `push` / `pop` — с правым концом
- `unshift` / `shift` — с левым концом (+ сдвиг всех элементов)
- Разница в производительности PACKED vs HOLEY для этих операций обусловлена тем, что V8 использует разные внутренние пути (code paths) для HOLEY-массивов — например, при `shift`/`unshift` нужно проверять, является ли каждый элемент дыркой при копировании. Но сам паттерн расположения дырок (регулярный vs случайный) и степень разреженности не меняют алгоритмическую сложность этих операций.

3. Когда случайное расположение имело бы смысл
Случайное расположение дырок было бы важно для бенчмарков:
- Итерации (`forEach`, `map`, `for...of`) — где предсказуемость паттерна может помочь branch predictor'у CPU
- Произвольного доступа по индексу — где V8 может по-разному оптимизировать dictionary mode vs sparse elements
- Очень высокой разреженности (>90% дырок) — V8 может переключить массив в `dictionary mode` (DICTIONARY_ELEMENTS), что кардинально меняет характеристики
