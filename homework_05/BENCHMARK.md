# Benchmark: сравнение реализаций PixelStream

**Платформа:** Node.js v24.13.0, macOS, tsx 4.21.0  
**Методология:** медиана 7 прогонов, 3 прогона прогрева (JIT warm-up) перед каждым замером

## Реализации

| Название | Внутреннее хранилище |
|---|---|
| **TypedArray** | `Uint8Array` длиной `width × height × 4` |
| **FlatArray** | `number[]` длиной `width × height × 4` |
| **Tuples** | `[r,g,b,a][]` — массив кортежей длиной `width × height` |
| **Objects** | `{r,g,b,a}[]` — массив объектов длиной `width × height` |

## Анализ

### TypedArray

**Создание:** `new Uint8Array(n)` — одна системная аллокация, не зависит от числа пикселей. При 2048×2048: **0.05 мс**.

**Запись:** 4 прямых присваивания `buffer[index+i] = rgba[i]` в `Uint8Array`. V8 компилирует каждое в `STORE_TYPED_ELEMENT` без проверок типа входного значения. При 2048×2048: **27 мс**.

**Чтение и forEach Row-major:** прямое чтение по индексу `buf[i]` + массивный литерал `[a,b,c,d]` — одна аллокация без вызовов методов. При 2048×2048: чтение **21 мс**, forEach **41 мс**.

**forEach Col-major:** шаг `width × 4` байт = 8 КБ при 2048×2048. Буфер `Uint8Array` хранится в contiguous memory, шаг значительно больше кэш-линии (64 байта), но байтовое хранение даёт наименьший stride среди всех реализаций. При 2048×2048: **58 мс**.

---

### FlatArray

**Создание:** `new Array(n).fill(0)` — резервирует capacity и заполняет нулями за один проход. При 2048×2048: **32 мс**.

**Запись:** 4 прямых присваивания `buffer[index+i] = rgba[i]` без цикла. При 2048×2048: **82 мс**.

**Чтение и forEach Row-major:** массивный литерал из 4 соседних элементов плоского буфера. При 2048×2048: чтение **23 мс**, forEach **44 мс**.

**forEach Col-major:** шаг `width × 4` элементов по `number[]` — каждый элемент 8 байт (JS number), итоговый шаг 16 КБ, каждый шаг — промах кэша. При 2048×2048: **186 мс**.

---

### Tuples

**Создание:** `Array.from({length: n}, () => [0,0,0,0])` создаёт `n` отдельных 4-элементных массивов на куче — по одной аллокации JSArray на пиксель. При 2048×2048: **272 мс**.

**Запись:** `buffer.at(index)` получает ссылку на кортеж (с неявной проверкой на отрицательный индекс), затем 4 прямых присваивания по индексу. При 2048×2048: **407 мс**.

**Чтение и forEach Row-major:** массивный литерал из 4 элементов существующего кортежа. При 2048×2048: чтение **22 мс**, forEach **48 мс**.

**forEach Col-major:** шаг `width` указателей = 16 КБ при 2048×2048. Кортежи — отдельные heap-объекты, разбросанные по памяти, каждый шаг по столбцу — промах кэша. При 2048×2048: **284 мс**.

---

### Objects

**Создание:** `Array.from({length: n}, () => ({r:0,g:0,b:0,a:0}))` — `n` объектов с единым hidden class. При 2048×2048: **321 мс**.

**Запись:** 4 прямых присваивания в именованные поля объекта с единым hidden class. V8 генерирует оптимизированный код доступа к полям. При 2048×2048: **325 мс**.

**Чтение и forEach Row-major:** деструктуризация `{r,g,b,a}` + массивный литерал. При 2048×2048: чтение **18 мс**, forEach **52 мс**.

**forEach Col-major:** шаг `width` указателей = 16 КБ при 2048×2048. При 2048×2048: **223 мс**.

## Итоговые выводы

| | TypedArray | FlatArray | Tuples | Objects |
|---|---|---|---|---|
| **Создание, 2048×2048** | **0.05 мс** | 32 мс | 272 мс | 321 мс |
| **Запись, 2048×2048** | **27 мс** | 82 мс | 407 мс | 325 мс |
| **Чтение, 2048×2048** | 21 мс | 23 мс | 22 мс | **18 мс** |
| **forEach Row, 2048×2048** | **41 мс** | 44 мс | 48 мс | 52 мс |
| **forEach Col, 2048×2048** | **58 мс** | 181 мс | 284 мс | 223 мс |

**TypedArray** — создание 0.05 мс при 2048×2048: `new Uint8Array(n)` делает одну системную аллокацию. Запись — 27 мс: 4 прямых `STORE_TYPED_ELEMENT` без проверок типа. Чтение и Row-major обход — 21 и 41 мс. Col-major обход — 58 мс: плоский байтовый буфер имеет stride 8 КБ при шаге по столбцу.

**FlatArray** — создание 32 мс, запись 82 мс при 2048×2048: `new Array(n).fill(0)` инициализирует буфер за один проход, 4 прямых присваивания по индексу. Чтение и Row-major обход — 23 и 44 мс. Col-major обход — 181 мс: stride 16 КБ (8 байт на элемент × 2048), каждый шаг — промах кэша.

**Tuples** — создание 272 мс при 2048×2048: по одной аллокации JSArray на пиксель. Запись — 407 мс: `buffer.at(index)` добавляет неявную проверку на отрицательный индекс перед каждой записью в кортеж. Чтение и Row-major обход — 22 и 48 мс. Col-major обход — 284 мс: кортежи — независимые heap-объекты, stride 16 КБ, каждый шаг — промах кэша.

**Objects** — создание 321 мс, запись 325 мс при 2048×2048: `n` объектов с единым hidden class, 4 присваивания в именованные поля. Чтение — 18 мс: деструктуризация `{r,g,b,a}` из объекта с единым hidden class — предсказуемый паттерн для JIT. Row-major обход — 52 мс. Col-major обход — 223 мс: stride 16 КБ, промах кэша на каждый шаг.
