# Бенчмарк: `at()` — произвольный доступ

## Описание

Сравнение производительности метода `at()` для двух реализаций бинарной сериализации массива строк:

|          Реализация          |                       Формат буфера                        |               Сложность `at(i)`               |
| :--------------------------: | :--------------------------------------------------------: | :-------------------------------------------: |
|  `hw_01` — `StringsBuffer`   |   `[size][len₁][data₁][len₂][data₂]…` (последовательный)   | **O(n)** — линейный проход до нужного индекса |
| `hw_02` — `StringsBufferImp` | `[len₁][ptr₁][len₂][ptr₂]…[data₁][data₂]…` (с указателями) |     **O(1)** — прямой доступ по смещению      |

## Методология

- Генерируются массивы случайных UTF-8 строк средней длиной ~20 символов (включая кириллицу)
- Для каждого размера массива выполняется **10 000** вызовов `at()` по **случайным** индексам
- Перед замером — прогрев (1 000 вызовов)
- Индексы предгенерированы, чтобы исключить влияние `Math.random()` на замер

**Окружение:** Node.js, macOS

## Анализ

### Ключевые наблюдения

1. **`hw_02` стабильно быстр** — время одного вызова `at()` составляет ~270–290 ns независимо от размера массива, что подтверждает **O(1)** сложность.

2. **`hw_01` деградирует линейно** — при увеличении массива в 100 раз (100 → 10 000) среднее время `at()` растёт примерно в 87 раз (3 591 → 312 123 ns/op), что соответствует **O(n)** сложности (среднее обращение к элементу на позиции ~n/2).

3. **Разрыв растёт с размером данных**:
   - На 100 элементах — ×12
   - На 1 000 элементах — ×116
   - На 10 000 элементах — **×1 137**

### Почему так?

|                         |                                    `hw_01` (sequential)                                     |                            `hw_02` (pointers)                             |
| :---------------------: | :-----------------------------------------------------------------------------------------: | :-----------------------------------------------------------------------: |
| Чтобы прочитать `at(i)` | Нужно пройти все `i` предшествующих записей, читая длину каждой и перепрыгивая через данные | Вычислить смещение `i * 8`, прочитать длину и указатель, перейти к данным |
|    Формула смещения     |                    Неизвестна заранее (зависит от длин предыдущих строк)                    |                    `offset = i × (4 + 4)` — константа                     |

### Вывод

Формат с указателями (`hw_02`) обеспечивает **константное время произвольного доступа** ценой дополнительных 4 байт на каждую строку (указатель). Это классический компромисс **пространство ↔ время**: небольшой overhead по памяти даёт кардинальное ускорение на операциях чтения по индексу.

## Запуск

node --experimental-strip-types hw_02/benchmark.mts
